## 1. タスクタイプの定義

### 採用するタイプ（2種 ＋ 将来1種）

| タイプ | 記号 | DB値 | 定義 | 追加方法 |
|--------|------|------|------|----------|
| **クエスト** | 🎯 | `mission` | 長期的な目標・挑戦。意志でやり続けるもの | LISTモーダル / LINEから追加後に変更 |
| **タスク** | ✅ | `default` / `appointment`（後方互換） | それ以外すべて。LINEからのデフォルト。旧スケジュールも統合 | LINE送信 / LISTモーダル |
| **ミッション** *(将来)* | 🔥 | `operator` | 運営側が提供する週次チャレンジ | 運営管理画面から配信 |

> **決定事項（2026-03-19）**: スケジュールタイプは廃止。タスクに統合。

### タイプの性質の違い

```
                 緊急度
                  高 ↑
      ┌──────────────┬──────────────┐
重要  │  スケジュール  │  タスク(⚡)   │
度   │  (時間決定済)  │  (緊急フラグ)  │
高   ├──────────────┼──────────────┤
↑   │  クエスト      │  タスク       │
    │  (重要×非緊急) │  (普通)       │
    └──────────────┴──────────────┘
                              → 緊急度低
```

## 2. タスクに付く属性

### 必須
- `task_name`: タスク名（文字列）
- `task_type`: クエスト / タスク / スケジュール

### タイプ別の追加属性

**クエスト:**
- `reason`: 「なぜやるの？」（任意）
- `target_date`: 「いつやる？/目標日」（任意）← NEW
- `urgent_flag`: ⚡今日中フラグ（boolean）

**タスク:**
- `urgent_flag`: ⚡今日中フラグ（boolean）
- `reason`: 任意

**スケジュール:**
- `scheduled_at`: 日時（必須）
- `location`: 場所（任意、将来）

## 3. 優先度システムの刷新

### 現状 → 新設計

| 現状 | 新設計 | 理由 |
|------|--------|------|
| 最重要 (critical) | クエスト + ⚡緊急フラグ | タイプで性質、フラグで緊急度を表現 |
| 重要 (high) | クエスト or スケジュール | 重要なものはタイプで表現できる |
| 普通 (normal) | タスク | 普通のTODO |
| ～（廃止）～ | ⚡今日中フラグ | タイプを問わず緊急なものをマーク |

### 移行方針
- DB上の `priority_level` は残す（後方互換）
- UIからは段階的に廃止し、⚡フラグに一本化
- critical → クエスト + 緊急フラグ に自動マイグレーション（要確認）

## 4. LINEキャプチャフロー

```
① ユーザーがLINEでテキスト送信
     ↓
② タイプ「タスク」として即保存（摩擦ゼロ）
     ↓  Bot返信: 「✅ 追加しました！いつやりますか？」+ クイックリプライ
③ クイックリプライ選択（今夜21時/明日朝8時/来週月曜/設定しない）
     ↓
④ LIST画面でタスクカードを開いて詳細設定（タイプ変更・理由入力等）
```

**原則: キャプチャ（記録）と分類（整理）は分離する（GTD）**

## 8. 実装優先度

### Phase 1（今すぐ）
1. タスクタイプ名変更: mission→クエスト, appointment→スケジュール, default→タスク
2. 「いつやる？」フィールドをクエストに追加
3. 追加モーダルを1画面に統合（タイプ+フィールド+緊急フラグ）
4. LIST表示をセクション分け（⚡今日中 / クエスト / スケジュール / タスク）

### Phase 2（中期）
1. 優先度システムの廃止（⚡フラグに一本化）
2. クエストへの進捗/サブステップ
3. 週次スタート時の「クエスト設定プロンプト」

### Phase 3（将来）
1. 運営ミッション機能
2. ソーシャル要素
3. フレキシブルストリーク
